# Migration Checklist

Use this for existing projects.

> For an unattended/agent-driven rollout with mandatory pause gates, see
> `scripts/convex-vercel/docs/ROLLOUT.md`. This checklist is the human-friendly
> sibling; the runbook is its agent-friendly equivalent.

## 1. Inventory

```bash
SETUP=/Users/dannywannagat/Code/coding-agent-setup/scripts/convex-vercel
$SETUP/cvv-audit.sh --project <project>
```

Record:

- Vercel project/root from `.vercel/project.json`
- Convex production and shared development deployment names
- Convex cloud/site URLs
- Modal prod/preview app names, if any
- current Vercel env names for production and preview
- current Convex env names for production and preview
- all public Convex functions and their auth boundary

## 2. Choose Profile

- Use `direct-convex` only when realtime browser Convex access is required and
  Convex functions enforce auth themselves.
- Use `server-proxy` for internal tools where PIN/session protection lives in
  Next.js and Convex should not be directly callable.
- Use `modal-proxy` when Modal endpoints are part of the backend.

## 3. Add Contract

```bash
$SETUP/cvv-scaffold.sh --project <project> --profile <profile>
```

Review dry-run output, then:

```bash
$SETUP/cvv-scaffold.sh --project <project> --profile <profile> --apply
```

Edit placeholders in:

- `docs/deployment.config.json`
- `docs/ENV.md`
- `.env.example`
- deploy scripts, especially project-specific lint/test commands and nested app
  roots

## 4. Normalize Env

Plan first:

```bash
$SETUP/cvv-env-plan.sh --project <project> --env production --target all
$SETUP/cvv-env-plan.sh --project <project> --env preview --target all
```

Check local readiness before writing cloud env, especially on a fresh Mac:

```bash
$SETUP/cvv-doctor.sh --project <project> --target all
```

Apply only after explicit approval:

```bash
$SETUP/cvv-env-apply.sh --project <project> --env production --target vercel --secrets-file <project>/.deploy/secrets.env --apply
$SETUP/cvv-env-apply.sh --project <project> --env preview --target vercel --secrets-file <project>/.deploy/secrets.env --apply
$SETUP/cvv-env-apply.sh --project <project> --env production --target convex --secrets-file <project>/.deploy/secrets.env --apply
$SETUP/cvv-env-apply.sh --project <project> --env preview --target convex --secrets-file <project>/.deploy/secrets.env --apply
```

Vercel apply overwrites existing environment variables by default.
`--target all --apply` validates both Vercel and Convex first, so missing or
wrong Convex deploy keys block all writes before Vercel is changed.

Preview Vercel env writes are branch-scoped by default. Run the command from the
target preview branch, or set `CVV_VERCEL_PREVIEW_BRANCH=<branch>` explicitly.

For direct Convex custom JWT auth, verify the JWKS URL before browser testing.
If `CONVEX_AUTH_JWKS_URL` points at production, production must already include
`/api/auth/jwks`. If the current feature branch is the first deployment adding
that route, Convex Preview needs a temporary reachable preview JWKS URL instead.
When Vercel Deployment Protection is enabled, use Vercel Protection Bypass for
Automation and put the bypassed JWKS URL only into Convex Preview env.

## 4a. Generate Local Secrets Without LLM Exposure

Prefer the project wrapper when scaffolded:

```bash
./scripts/cloud-env.sh secrets --env all --generate-missing --apply
./scripts/cloud-env.sh apply --env preview --target all --apply
./scripts/cloud-env.sh apply --env production --target all --apply
```

Or call the central scripts directly:

```bash
$SETUP/cvv-secrets.sh --project <project> --env all --generate-missing --apply
$SETUP/cvv-env-apply.sh --project <project> --env preview --target all --secrets-file <project>/.deploy/secrets.env --apply
```

Secrets are written to `.deploy/secrets.env` with mode 600 and must remain
gitignored. The scripts refuse to generate or apply a project-local secrets file
unless Git ignores it. Convex deploy keys are not generated by these scripts;
copy them manually from the Convex Dashboard and add them only to `.env` or
`.deploy/secrets.env` as `prod:<deployment>|<token>` and
`dev:<deployment>|<token>`. Agents may verify key names and script output, but
must not print or copy values.

Rotation:

```bash
./scripts/cloud-env.sh secrets --env preview --rotate app-proxy --apply
./scripts/cloud-env.sh secrets --env production --rotate jwt --apply
./scripts/cloud-env.sh apply --env preview --target all --apply
```

## 5. Harden API Access

- Next.js API routes must validate session/PIN/API secret before expensive work.
- Convex public functions must enforce auth or become internal functions.
- Convex HTTP actions must verify shared secrets with timing-safe compare.
- Modal endpoints must verify a shared secret before executing jobs.
- Missing PIN/session/secret in production or preview should fail closed.

## 6. Verify

```bash
$SETUP/cvv-audit.sh --project <project> --strict
```

Then run project-specific lint, tests, typecheck, and security tests proving
unauthenticated direct API/Convex/Modal calls fail.
